\section{Task Cost Model}
\label{sec:task-cost-model}

Section~\ref{sec:problem-statement} states that task profiling produces several
classes of outputs: cost estimates, reliability diagnostics, risk
interpretation, and profiling-policy recommendations. This section defines only
the first class. The cost model estimates attempt-attributed active resource
demand and worker occupancy. It does not define uncertainty metrics, risk
scoring, graph topology, raw-observation mapping, or policy selection.

\subsection{Attempt-Level Cost Object}
\label{sec:cost-attempt-level}

The cost model is defined at task-attempt granularity. A task kind determines
the reusable model structure, a task instance supplies intrinsic work features,
and a task attempt is the concrete execution in which demand, occupancy,
observations, outcomes, and residuals are realized.

Let $a \in \mathcal{A}$ be a task attempt. The attempt executes task instance
$i(a) \in \mathcal{I}$, whose task kind is $k(i(a)) \in \mathcal{K}$. Write
$k_a = k(i(a))$. The cost estimate for attempt $a$ is
\[
\widehat{\mathbf{C}}(a)
=
\left(
\widehat{\mathbf{D}}(a),
\widehat{\mathbf{U}}(a)
\right).
\]
The vector $\widehat{\mathbf{D}}(a)$ contains resource-demand estimate objects,
and $\widehat{\mathbf{U}}(a)$ contains worker-occupancy estimate objects. The
cost object has no universal scalar value.

For attempt $a$,
\[
\widehat{\mathbf{D}}(a)
=
\left\{
\widehat{E}_r(a)
\mid
r \in \mathcal{R}_{k_a}
\right\},
\qquad
\widehat{\mathbf{U}}(a)
=
\left\{
\widehat{E}_o(a)
\mid
o \in \mathcal{O}_{k_a}
\right\}.
\]
Point values and quantiles are projections from these estimate objects, not
separate cost objects.

\subsection{Resource Demand and Worker Occupancy}
\label{sec:cost-demand-occupancy}

Resource demand is the estimated amount of active resource use required by, or
expected for, an attempt along a resource dimension. Observed resource usage is
the measured attempt-attributed use of that resource when attribution is
available. Worker occupancy is bounded execution capacity held by an attempt
over time.

For an occupancy dimension $o$, let $\mathcal{I}_{o,a}$ be the set of intervals
during which the capacity associated with $o$ is held by attempt $a$. If the
attempt can hold more than one unit of that capacity, let $m_o(t,a)$ denote the
amount held at time $t$. An observed occupancy value can be understood as
\[
Y_o(a)
=
\int_{\bigcup \mathcal{I}_{o,a}}
m_o(t,a)\,dt,
\qquad
o \in \mathcal{O}_{k_a}.
\]
This expression states the semantics: occupancy is capacity held over time, not
elapsed wall-clock time by itself.

Execution context, pressure, waiting, and elapsed time are not automatically
cost dimensions. CPU saturation on a node is context, not CPU demand of the
attempt. Worker-pool pressure is context, not worker-slot time. Dependency wait
time is an observed waiting signal. It increases worker-slot occupancy only when
the worker slot remains held during the wait.

\subsection{Feature Projections and Model State}
\label{sec:cost-feature-projections-model-state}

Let $z_a \in \mathcal{Z}_{k_a}$ be the canonical input-feature vector available
for attempt $a$ after feature mapping and normalization. It may be partial and
may include explicit missingness indicators. It contains input features used to
estimate the attempt; it does not contain post-execution observed output values
for the same attempt.

Two task-kind-specific projections separate the roles of canonical input
features:
\[
\mathbf{x}_a = P_X^{k_a}(z_a),
\qquad
\mathbf{c}_a = P_C^{k_a}(z_a).
\]
The projection $\mathbf{x}_a \in \mathcal{X}_{k_a}$ contains intrinsic task
features. The projection $\mathbf{c}_a \in \mathcal{C}_{k_a}$ contains
execution-context features. The reusable task-kind model state is
$\theta_{k_a} \in \Theta_{k_a}$.

\subsection{Typed Cost Dimensions}
\label{sec:cost-typed-dimensions}

A cost dimension is not a metric name. It is a typed descriptor:
\[
\Delta_q
=
\left(
\rho_q,
u_q,
\mathcal{V}_q,
\alpha_q,
\omega_q,
\kappa_q
\right).
\]
Here $\rho_q$ is the role, $u_q$ is the unit, $\mathcal{V}_q$ is the value
domain, $\alpha_q$ is the attribution boundary, $\omega_q$ is the observation
semantics, and $\kappa_q$ is the composition operator.

CPU time usually composes by summation over attributed execution intervals.
Memory peak usually composes by maximum. Worker-slot time composes over
intervals during which the slot is held. Connection hold time composes over
connection-acquire and connection-release intervals. These dimensions should not
be forced into one aggregation rule.

\subsection{Dimension Estimate Objects}
\label{sec:cost-dimension-estimate-objects}

For each cost dimension $q$, let $\mathcal{E}_q$ be the estimate-object space.
The dimension estimate for attempt $a$ is
\[
\widehat{E}_q(a) \in \mathcal{E}_q.
\]
The point estimate is
\[
\widehat{Y}_q(a)
=
\operatorname{point}_q\!\left(\widehat{E}_q(a)\right).
\]
When distributional information is available,
\[
\widehat{Y}_{q,p}(a)
=
\operatorname{quantile}_{q,p}\!\left(\widehat{E}_q(a)\right),
\qquad
p \in (0,1).
\]

\subsection{Dimension-Specific Estimators}
\label{sec:cost-dimension-specific-estimators}

For each task kind $k$ and dimension $q \in \mathcal{Q}_k$, the model defines
\[
F_{k,q}
:
\mathcal{X}_k
\times
\mathcal{C}_k
\times
\Theta_k
\rightarrow
\mathcal{E}_q.
\]
For attempt $a$,
\[
\widehat{E}_q(a)
=
F_{k_a,q}\!\left(
\mathbf{x}_a,
\mathbf{c}_a;
\theta_{k_a}
\right).
\]
The full estimator is the collection $F_k=\{F_{k,q}\mid q\in\mathcal{Q}_k\}$.

Some dimensions may use a component model:
\[
\widehat{E}_q(a)
=
\operatorname{estimate}_q\!\left(
\widehat{B}_q(k_a,\mathbf{x}_a;\theta_{k_a}),
\widehat{A}_q(k_a,\mathbf{x}_a,\mathbf{c}_a;\theta_{k_a}),
\widehat{O}_q(k_a,\mathbf{x}_a,\mathbf{c}_a;\theta_{k_a}),
\kappa_q
\right).
\]
For dimensions that permit proportional amplification plus additive overhead,
the point projection may use
\[
\widehat{Y}_q(a)
=
\widehat{B}_q(k_a,\mathbf{x}_a;\theta_{k_a})
\cdot
\widehat{A}_q(k_a,\mathbf{x}_a,\mathbf{c}_a;\theta_{k_a})
+
\widehat{O}_q(k_a,\mathbf{x}_a,\mathbf{c}_a;\theta_{k_a}).
\]
This affine form is a specialization, not a universal law.

\subsection{Execution Overhead and Profiling Overhead}
\label{sec:cost-overhead-types}

Execution overhead is overhead required by task execution itself: setup,
deserialization, connection establishment, commit, acknowledgement, and cleanup.
Profiling overhead is overhead introduced by observation policy: tracing,
sampling, runtime profiling, high-cardinality labels, telemetry export, and
additional accounting. Both can contribute to cost dimensions, but neither is a
third output family.

A base estimate excludes additional profiling overhead:
\[
\widehat{\mathbf{C}}_{\mathrm{base}}(a).
\]
A policy-aware estimate includes the overhead of an already selected profiling
policy $\rho$:
\[
\widehat{\mathbf{C}}_{\rho}(a).
\]

\subsection{Observed Values and Residuals}
\label{sec:cost-residuals-usage}

For $q \in \mathcal{Q}_{k_a}$, the observed value $Y_q(a)$ is not raw telemetry.
It is an attempt-attributed observed value produced according to $\omega_q$.
The residual is defined only when an observed value exists:
\[
\epsilon_q(a)
=
Y_q(a) - \widehat{Y}_q(a).
\]
Absence of a residual does not imply model accuracy; it may only mean that the
dimension was not observed.

\subsection{Distributional Cost Estimates}
\label{sec:cost-distributional-estimates}

Attempt-level cost may be variable even within one task kind and feature bucket.
The estimate-object formulation allows each dimension to carry distributional
information such as histograms, quantile sketches, conservative envelopes, or
prediction intervals. This section does not define confidence or risk metrics;
later sections derive diagnostics from residuals, tails, drift, missingness,
multimodality, and insufficient history.

\subsection{Boundaries of the Cost Model}
\label{sec:cost-model-boundaries}

The cost model defines active resource demand and worker occupancy. It does not
define influence domains, graph topology, raw-observation mapping, uncertainty
metrics, risk scoring, or profiling-policy selection. Those layers are defined
in later sections.
